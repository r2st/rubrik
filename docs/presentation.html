Transcript Intelligence

An auto-scalable system for topic categorization, sentiment, and strategic insights from B2B meeting transcripts.
Target scale: millions to 100M+ records.

Auto-scalable FastAPI scikit-learn Gemma 4 (QLoRA) + frontier cascade Postgres + Iceberg + Kafka Ray Train + vLLM 14 ADRs 345 tests · 94% cov

Take-home submission · Rubrik · 2026

The brief

Build an auto-scalable system that turns raw meeting transcripts — at production volume of millions to 100M+ records — into actionable signal.

Inputs (per meeting)

  • Title + metadata
  • Full transcript (sentences with sentiment labels)
  • Summary, action items, speakers, key topics

Required signal

  • Topic categorization (call type · purpose · product)
  • Sentiment trajectories — not just averages
  • Strategic insights — risk, incidents, churn, competition

Constraints I imposed on myself

  • Auto-scalable by design. Every component states its scale envelope and the substrate it switches to past that envelope.
  • Production-shaped from day 1 — no notebook-only artifacts.
  • Production volume — not a development snapshot — is the architecture target.
  • Every non-trivial decision recorded as an ADR.

Why this matters

  • Trajectory-based signals beat averaged sentiment on early-warning detection.

  • Validation is semantic, not just unit-test — validate.py audits rules against the actual dataset.

Hybrid by design. Each layer handles what it's best at.

Auto-scaling the API tier

15 specific gaps closed in code (ADR 0013). Highlights:

Cold start fixed

Singleton CronJob writes PipelineState snapshot to S3. Replicas warm in seconds, not minutes.

Cluster-wide rate limit

slowapi storage on Redis. Per-tenant fairness via (tenant_id, ip) key. Per-tenant cap overrides via FastAPI dependency.

HPA on real signals

RPS + p95 latency via Prometheus Adapter. CPU only as guardrail.

Backpressure + circuit breakers

Inflight cap → 503 + Retry-After. readiness_db_probe breaker stops cascading failure.

Graceful shutdown

Lifespan flips readiness, sleeps, drains tasks, disposes pool. Zero-drop rolling deploys.

Push invalidation

Postgres LISTEN/NOTIFY on settings_changed — operator changes propagate <100ms instead of 5s TTL.

Plus: alembic-as-Job, PgBouncer, split /api/live ↔ /api/ready, Arq queue, OTel head + tail sampling, CDN runbook.

Production readiness — what's actually wired

Security

  • API-key auth (DB-backed) · JWT (opt-in, JWKS) · TOTP MFA + backup codes
  • PBKDF2-SHA256 admin pw + file-mountable session secret
  • CSRF (double-submit cookie + Sec-Fetch-Site)
  • 5/min/IP login + 60/min/IP admin-write throttles
  • 1 MiB body cap · Idempotency-Key replay-safe writes
  • PII redactor (egress) + PII filter on logs
  • secret type masked everywhere; GDPR delete (structured match)
  • Per-tenant data isolation via ContextVar (JWT claim → repo + cache)
  • Snapshot HMAC signing closes the pickle-RCE primitive
  • Streaming bulk export with per-day quota; audit rows carry IP + UA
  • CSP / HSTS / CORS (default [], strip * in prod) · robots.txt + security.txt

Resilience & caching

  • Three-layer rate limit (global + concurrency + adaptive)
  • Circuit breakers (state metric)
  • Multi-tier cache: HTTP ETag · entity Redis · LLM response · search query
  • Cache-stampede protection (TTL jitter + single-flight + SWR + negative cache)
  • Outbox + DLQ + admin replay endpoint
  • Graceful drain shutdown · Cold-start snapshot · Last-good-state

Observability

  • Structured logs (JSON in prod) · X-Request-ID end-to-end
  • Prometheus /metrics + custom series (outbox / idempotency / throttle / breaker)
  • SLO burn-rate alerts (1h@14.4× · 6h@6×)
  • OTel head + tail sampling · trace_id propagated through outbox
  • Live + Ready probes (DB + Redis + breaker)
  • Sentry (opt-in)

What I deliberately didn't do

  • No LLM in categorization. Rules cover ~90% at <1ms; zero-shot LLM matched accuracy but added cost, latency, and a non-determinism surface. ADR 0002.
  • No JWT yet. Single-tenant, single-consumer. JWT is a one-line dependency swap when multi-tenancy lands. ADR 0006.
  • No DB for analytical state. The pipeline cache is per-replica + a shared snapshot. Adding a warehouse is the ADR 0008 path; today the snapshot is the bridge.
  • No streaming clustering. Streaming pipeline skips clustering + viz; those run separately at scale (Spark MLlib / sampled). ADR 0011.
  • Frontier-LLM gateway scaffold shipped, provider HTTP layer stubbed. ADR 0012 — gateway, PII redactor, circuit breaker, daily $ budget, audit log all wired; production deploys plug in the real anthropic / openai / google client where the scaffold raises NotImplementedError.
  • Kafka client class shipped; broker is deploy-time. KafkaPublisher in api/outbox_kafka.py + Strimzi manifest + KEDA scaler + outbox DLQ wrapper are all in place; pointing at a real broker is one URL in bootstrap.toml.
  • DatabaseRepository + alembic migration ship together. transcripts.repository = database flips backend at runtime; alembic upgrade head creates the meetings table with the tenant_id column (migration 0004).
  • Tenant ContextVar plumbed, JWT issuance isn't. The read path is tenant-scoped end-to-end (repo + cache). Token issuance is still operator-side via an external IDP — out of scope for a single-admin take-home.
  • DR runbook documents cross-region; no live drill. Quarterly drill is the next step. The PITR + 35-day backup + monthly restore drill are wired.
  • No SOC-2 certification. Control inventory exists in SECURITY.md + the security architecture slide; audit doesn't.
