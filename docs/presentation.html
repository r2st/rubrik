Transcript Intelligence

An auto-scalable system for topic categorization, sentiment, and strategic insights from B2B meeting transcripts.
Target scale: millions to 100M+ records. The dataset shipped with the brief is a representative sample used for verification.

Auto-scalable FastAPI scikit-learn Gemma 4 (QLoRA) Postgres + Iceberg Ray Train + vLLM 183 tests · 94% cov

Take-home submission · Rubrik · 2026

The brief

Build an auto-scalable system that turns raw meeting transcripts — at production volume of millions to 100M+ records — into actionable signal. The dataset shipped with the brief is a representative sample used for end-to-end verification.

Inputs (per meeting)

  • Title + metadata
  • Full transcript (sentences with sentiment labels)
  • Summary, action items, speakers, key topics

Required signal

  • Topic categorization (call type · purpose · product)
  • Sentiment trajectories — not just averages
  • Strategic insights — risk, incidents, churn, competition

Constraints I imposed on myself

  • Auto-scalable by design. Every component states its scale envelope and the substrate it switches to past that envelope.
  • Production-shaped from day 1 — no notebook-only artifacts.
  • The shipped dataset is a sample for verification, not the architecture target.
  • Every non-trivial decision recorded as an ADR.

Headline findings (verified on the sample)

At-risk customers

2

Composite-score 🔴 high-risk (sentiment gap + churn signals + negative pivots).

Latent themes (k=7)

7

Silhouette-picked TF-IDF clusters. Catches multi-product migrations the rules can't see.

Neg-pivot meetings

12

Ended OK on average but had a sharp mid-meeting drop — friction events.

Why this matters

  • Trajectory-based signals beat averaged sentiment on early-warning detection.
  • Hybrid categorization catches both explicit (regex on titles) and latent (KMeans on bodies) patterns.
  • Validation is semantic, not just unit-test — validate.py audits rules against the actual dataset.

Analytics core — rules + ML + LLM

Hybrid by design. Each layer handles what it's best at.

LayerAlgorithmWhereWhy this layer
Rules Compiled regex + keyword bags (config-driven) src/categorizer.py ~90% coverage, <1ms, deterministic, fully auditable.
Sentiment Per-sentence labels + numpy trajectory math src/sentiment.py Catches mid-meeting pivots an averaged score hides.
Classical ML TF-IDF → KMeans (silhouette-picked k) src/clustering.py Latent cross-cutting themes the rules' fixed taxonomy can't see.
LLM tier 1 Gemma 4 E4B-it + QLoRA (ROUGE-L 0.39) gemma-finetune/ Generative work — summaries, action items — at self-hosted economics.
LLM tier 2 Frontier API (Claude / GPT-4 / Gemini) Gateway · ADR 0012 ~5% escalation: OOD, long context, world knowledge, high-stakes.

No LLM in the categorization path — rules already solve it cheaply, deterministically, on-perimeter.

Auto-scaling the API tier

15 specific gaps closed in code (ADR 0013). Highlights:

Cold start fixed

Singleton CronJob writes PipelineState snapshot to S3. Replicas warm in seconds, not minutes.

Cluster-wide rate limit

slowapi storage on Redis. Per-tenant fairness via (tenant_id, ip) key. Per-tenant cap overrides via FastAPI dependency.

HPA on real signals

RPS + p95 latency via Prometheus Adapter. CPU only as guardrail.

Backpressure + circuit breakers

Inflight cap → 503 + Retry-After. readiness_db_probe breaker stops cascading failure.

Graceful shutdown

Lifespan flips readiness, sleeps, drains tasks, disposes pool. Zero-drop rolling deploys.

Push invalidation

Postgres LISTEN/NOTIFY on settings_changed — operator changes propagate <100ms instead of 5s TTL.

Plus: alembic-as-Job, PgBouncer, split /api/live ↔ /api/ready, Arq queue, OTel head + tail sampling, CDN runbook.

Production readiness — what's actually wired

Security

  • API-key auth (DB-backed)
  • PBKDF2-SHA256 admin pw
  • HMAC session cookie
  • 5/min/IP login throttle
  • 1 MiB body cap (DoS)
  • CSP / HSTS / CORS / etc.
  • Audit log on every write

Resilience

  • Backpressure (503 shed)
  • Circuit breakers
  • Graceful drain shutdown
  • Last-good-state on refresh fail
  • X-State-Age + Stale headers
  • Cold-start snapshot

Observability

  • Structured logs (JSON in prod)
  • X-Request-ID end-to-end
  • Prometheus /metrics
  • OTel head + tail sampling
  • Sentry (opt-in)
  • Live + Ready probes

Throughput baseline

Local load test (11 endpoints, weighted mix): ~395 RPS · p95 ≤ 35 ms.

What I deliberately didn't do

  • No LLM in categorization. Rules cover ~90% at <1ms; zero-shot LLM matched accuracy but added cost, latency, and a non-determinism surface. ADR 0002.
  • No JWT yet. Single-tenant, single-consumer. JWT is a one-line dependency swap when multi-tenancy lands. ADR 0006.
  • No DB for analytical state. The pipeline cache is per-replica + a shared snapshot. Adding a warehouse is the ADR 0008 path; today the snapshot is the bridge.
  • No streaming clustering. Streaming pipeline skips clustering + viz; those run separately at scale (Spark MLlib / sampled). ADR 0011.
  • No frontier-model gateway shipped yet. Designed (ADR 0012) but not built — Tier-1 fine-tune is sufficient on the sample.
